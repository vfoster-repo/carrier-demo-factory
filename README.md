# Carrier Demo Factory

An autonomous agent pipeline that turns a folder of trucking company CSVs into a
complete sales demo — a live insights report, a company profile page, and a working
operations platform — with no human steps between input and output.

Built with [Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview).

---

## The Problem It Solves

Data consultancies targeting small trucking carriers face a chicken-and-egg problem:
building a custom demo takes weeks, but carriers won't commit without seeing something
real first.

The Carrier Demo Factory breaks that loop. Drop in a carrier's CSV exports, run the
pipeline, and in a few hours you have three production-quality deliverables to put
in front of the prospect:

| Deliverable | What it is | Purpose |
|---|---|---|
| `report.html` | Self-contained insights report with interactive charts | Free hook — shows them exactly what's wrong with their operation |
| `profile.html` | Company profile / case study page | Context for everyone else who sees the demo |
| `app.py` | Multi-page Streamlit operations platform | The paid product — a prototype of the SaaS you'd build for them |

All three are generated autonomously from one input: a folder of the company's CSV files.

## How the Pipeline Works

```
[Optional] Carrier CSV exports, OR run Phase 0 to generate synthetic data
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 0 — Demo Data Generator  (01-company-creator.md)     │
│  OPTIONAL. Designs a fictional carrier, generates synthetic │
│  CSVs with embedded pain points. Skip when using real data. │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 1 — Idea Generator  (02-idea-generator.md)           │
│  Analyzes data like a consultant, independently discovers   │
│  the pain points, writes ideas.md with app specs            │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 2 — Report Builder  (03-report-builder.md)           │
│  Builds a polished self-contained HTML insights report      │
│  with embedded Plotly charts — no external dependencies     │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 3 — Profile Builder  (04-profile-builder.md)         │
│  Builds a company profile / case study page in dark         │
│  GitHub-style design — who they are, what was found,        │
│  what was built                                             │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 4 — Ops Platform Builder  (05-ops-builder.md)        │
│  Builds a full multi-page Streamlit operations platform     │
│  tailored to this company's specific pain points and data   │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
Output: report.html  +  profile.html  +  apps/ops/app.py
```

Each phase is a Claude Code agent prompt. The orchestrator (`scripts/run-cycle.sh`)
manages state, advances phases, and can run unattended on a VPS via a systemd timer.

---

## What Makes This Different

**The agent is not filling a template.** Each prompt instructs Claude to:

1. Read the raw data independently (Phase 2 is explicitly told *not* to read the pain points section of the company profile — it must discover them analytically)
2. Write company-specific prose, charts, and code that uses real names, real numbers, and real routes from the data
3. Self-review against a quality checklist before finishing

The result is output that feels hand-crafted for each company, not mail-merged.

## Requirements

- [Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview) with an active Anthropic API subscription
- Python 3.10+ with `streamlit`, `pandas`, `plotly` installed
- A Linux server with [Caddy](https://caddyserver.com/) for HTTPS + reverse proxy (for live deployment)
- `systemd` for the automated pipeline timer (optional — can also run manually)

See [docs/setup.md](docs/setup.md) for the full step-by-step setup guide.

## Quick Start (running the pipeline manually)

```bash
git clone https://github.com/vfoster-repo/carrier-demo-factory
cd carrier-demo-factory

# Configure
cp config.env.example config.env
# Edit config.env: set AGENT_DOMAIN, AGENT_EMAIL, AGENT_DIR, COMPANIES_DIR

# Drop a carrier's CSV files into a company directory
mkdir -p /your/companies/dir/my-carrier/data
cp /path/to/carrier/csvs/*.csv /your/companies/dir/my-carrier/data/

# Run the full pipeline (all phases in one session)
bash scripts/run-cycle.sh
```

Output lands in `COMPANIES_DIR/{company-slug}/`. Copy it into `demos/` to
commit to the repo and serve via GitHub Pages.

## Using Real Carrier Data

> **This is a prototype, not a finished product.** The demo data generator (Phase 0) exists
> so you can showcase the full pipeline without a real client. For an actual engagement,
> skip it entirely — the core pipeline (Phases 1–4) works identically on real data.

For a real prospect:

1. Get their CSV exports — loads, trips, trucks, drivers, fuel purchases, etc.
2. Create `{company-slug}/data/` and drop the CSVs in
3. Write a short `company_profile.md` (company name, fleet size, key customers, lanes)
4. Set `PHASE: generate_ideas` in `current_state.txt` and run the pipeline

Phases 2–5 are the actual product. The prompts instruct Claude to check real
column names before assuming anything, so the data doesn't need to match the
demo schema exactly.
